# Часто задаваемые вопросы (FAQ)

## Общие вопросы

### Q1: Зачем мигрировать с SourceCraft на GitHub?
**A**: Причины могут быть разными:
- Команда предпочитает экосистему GitHub
- Интеграция с определенными инструментами
- Корпоративные требования
- Консолидация репозиториев в одном месте
- Доступ к specific GitHub features

### Q2: Что такое зеркальная миграция?
**A**: Зеркальная миграция — это процесс создания точной копии репозитория, включая:
- Все коммиты и историю
- Все ветки и теги
- Все конфигурации Git
- Все notes и другие refs

### Q3: Безопасна ли миграция?
**A**: Да, если следовать лучшим практикам:
1. Всегда делайте backup перед миграцией
2. Тестируйте на тестовом репозитории
3. Проверяйте миграцию после завершения
4. Имейте план отката

### Q4: Сколько времени занимает миграция?
**A**: Зависит от:
- Размера репозитория
- Скорости интернет-соединения
- Сложности истории
- Типично: от 5 минут до нескольких часов

## Технические вопросы

### Q5: Что делать, если репозиторий очень большой (> 5GB)?
**A**: Для больших репозиториев:
```bash
# 1. Используйте shallow clone
git clone --mirror --depth 1 ИСТОЧНИК

# 2. Разбейте push на части
git push origin main develop --mirror
# Затем остальные ветки по одной

# 3. Используйте SSH (обычно быстрее)
git clone --mirror ssh://...

# 4. Рассмотрите очистку истории перед миграцией
git gc --aggressive --prune=now
```

### Q6: Как мигрировать репозиторий с подмодулями?
**A**: Подмодули нужно мигрировать отдельно:
```bash
# 1. Мигрируйте основной репозиторий
git clone --mirror --recurse-submodules ИСТОЧНИК

# 2. Для каждого подмодуля повторите процесс
# 3. После миграции обновите .gitmodules
# 4. Обновите URL подмодулей на GitHub
```

### Q7: Что происходит с защищенными ветками?
**A**: Защита веток не мигрируется автоматически. После миграции:
1. Настройте branch protection rules на GitHub
2. Настройте required status checks
3. Настройте required reviews
4. Протестируйте защиту

### Q8: Как мигрировать Git LFS файлы?
**A**: Git LFS требует дополнительных шагов:
```bash
# 1. Установите Git LFS
git lfs install

# 2. Клонируйте с LFS
git lfs clone --mirror ИСТОЧНИК

# 3. Настройте LFS на GitHub
# 4. Отправьте LFS объекты
git lfs push --all ЦЕЛЬ
```

## Проблемы и решения

### Q9: Ошибка "Authentication failed"
**Решение:**
```bash
# 1. Проверьте токен
echo $SOURCE_CRAFT_TOKEN

# 2. Проверьте срок действия токена
# 3. Обновите токен если нужно
# 4. Используйте правильный формат URL:
https://username:token@git.sourcecraft.dev/org/repo.git
```

### Q10: Ошибка "Repository not found"
**Решение:**
```bash
# 1. Проверьте URL
echo $SOURCE_URL

# 2. Проверьте права доступа
# 3. Убедитесь, что репозиторий существует
# 4. Проверьте видимость (public/private)
```

### Q11: Ошибка "Remote branch already exists"
**Решение:**
```bash
# 1. Используйте --force (осторожно!)
git push --mirror --force ЦЕЛЬ

# 2. Или удалите ветку на цели сначала
git push origin :refs/heads/ветка

# 3. Или переименуйте ветку перед миграцией
git branch -m старая-ветка новая-ветка
```

### Q12: Ошибка "RPC failed; HTTP 413"
**Решение:** Репозиторий слишком большой для одного push.
```bash
# 1. Разбейте на несколько push
git push origin main develop --mirror
sleep 10
git push origin feature/* --mirror

# 2. Увеличьте размер буфера Git
git config --global http.postBuffer 524288000  # 500MB

# 3. Используйте SSH
```

## После миграции

### Q13: Как проверить, что миграция прошла успешно?
**A**: Используйте скрипт проверки:
```bash
# Из папки проекта
./scripts/verify.sh

# Или вручную
git ls-remote ИСТОЧНИК | wc -l
git ls-remote ЦЕЛЬ | wc -l
diff <(git ls-remote ИСТОЧНИК) <(git ls-remote ЦЕЛЬ)
```

### Q14: Что делать с CI/CD пайплайнами?
**A**: После миграции:
1. Обновите URL репозитория в CI/CD конфигурациях
2. Обновите токены доступа
3. Протестируйте пайплайны
4. Обновите webhooks

### Q15: Как обновить локальные копии у разработчиков?
**A**: Разработчики должны:
```bash
# 1. Добавить новый remote
git remote add github НОВЫЙ_URL

# 2. Переименовать origin (опционально)
git remote rename origin sourcecraft

# 3. Установить github как origin
git remote set-url origin НОВЫЙ_URL

# 4. Получить обновления
git fetch --all
```

### Q16: Что делать со старым репозиторием на SourceCraft?
**A**: Рекомендуется:
1. **Архивировать**: Сделать read-only
2. **Перенаправлять**: Обновить README с ссылкой на GitHub
3. **Удалить**: Только после уверенности в успешной миграции
4. **Сохранить как backup**: На случай проблем

## Безопасность

### Q17: Как безопасно хранить токены?
**A**: Никогда не храните токены в репозитории:
```bash
# ПЛОХО: в .env файле в репозитории
# ХОРОШО: в .env файле в .gitignore
# ЛУЧШЕ: в переменных окружения системы
# ОТЛИЧНО: в secret manager (Hashicorp Vault, AWS Secrets Manager)
```

### Q18: Что делать, если токен скомпрометирован?
**A**: Немедленно:
1. Отзовите токен в настройках аккаунта
2. Создайте новый токен
3. Обновите все системы, использующие старый токен
4. Проверьте логи на подозрительную активность

### Q19: Как настроить двухфакторную аутентификацию?
**A**: Рекомендуется для обоих сервисов:
- **GitHub**: Settings → Security → Two-factor authentication
- **SourceCraft**: Настройки → Безопасность → 2FA
- Используйте приложение-аутентификатор (Google Authenticator, Authy)

## Производительность

### Q20: Как ускорить миграцию?
**A**: Оптимизации:
```bash
# 1. Используйте SSH вместо HTTPS
# 2. Настройте кэш Git
git config --global core.preloadindex true
git config --global core.fscache true

# 3. Используйте параллельные операции
git fetch --jobs=4

# 4. Очистите историю перед миграцией
git gc --aggressive
```

### Q21: Что делать при медленном интернет-соединении?
**A**: Стратегии:
1. Мигрируйте в непиковые часы
2. Используйте shallow clone
3. Разбейте миграцию на несколько сессий
4. Рассмотрите миграцию с физического носителя

### Q22: Как мигрировать несколько репозиториев?
**A**: Используйте скрипт:
```bash
#!/bin/bash
REPOS=("repo1" "repo2" "repo3")

for repo in "${REPOS[@]}"; do
    echo "Миграция $repo..."
    ./scripts/migrate.sh -s "sourcecraft/org/$repo" -t "github/user/$repo"
done
```

## Специфичные сценарии

### Q23: Как мигрировать организацию целиком?
**A**: Для миграции всей организации:
1. Получите список всех репозиториев
2. Создайте организацию на GitHub
3. Мигрируйте репозитории по одному
4. Настройте команды и права доступа
5. Обновите интеграции

### Q24: Что делать с wiki и issues?
**A**: Wiki и issues не мигрируются автоматически:
- **Wiki**: Экспортируйте вручную или используйте инструменты миграции
- **Issues**: Используйте GitHub Importer или сторонние инструменты
- **Pull Requests**: Обычно не мигрируются, только код

### Q25: Как мигрировать репозиторий с бинарными файлами в истории?
**A**: Для репозиториев с бинарными файлами:
1. Рассмотрите использование Git LFS
2. Очистите историю от больших бинарных файлов
3. Используйте `git filter-branch` или `git filter-repo`
4. Будьте готовы к долгой миграции

### Q26: Что делать с custom Git hooks?
**A**: Git hooks не копируются автоматически:
1. Скопируйте hooks из `.git/hooks/`
2. Добавьте их в репозиторий (например, в `scripts/git-hooks/`)
3. Настройте автоматическую установку hooks
4. Протестируйте hooks после миграции
